# Task: Set Up GitHub Actions Desktop Releases

## Goal

Set up GitHub Actions so this project is compiled on GitHub's servers and published as GitHub Release assets.

The user should only need to push source code and create a version tag. GitHub should build the compiled desktop app files automatically.

The project must build both:

- Windows version
- macOS version

Use the existing project scripts:

```txt
npm run desktop:dist:win
npm run desktop:dist:mac
```

Do not edit, rename, remove, or rewrite these existing scripts.

If extra scripts are genuinely needed, add new scripts only.

---

## What This Should Do

When the user pushes a git tag like:

```txt
v1.0.1
```

GitHub Actions should:

1. checkout the source code
2. install dependencies
3. run the Windows desktop build on a Windows runner
4. run the macOS desktop build on a macOS runner
5. create or update the GitHub Release for that tag
6. upload the compiled installers/build files to the release

The compiled app should be generated by GitHub Actions. The user should not manually upload the compiled app.

## Dependency Install Rules

Check which lockfile exists.

### If `package-lock.json` exists

Use:

```yml

  run: npm ci
```

### If `pnpm-lock.yaml` exists

Use pnpm instead:

```yml
- name: Setup pnpm
  uses: pnpm/action-setup@v4
  with:
    version: 9

- name: Setup Node
  uses: actions/setup-node@v4
  with:
    node-version: 22
    cache: pnpm

  run: pnpm install --frozen-lockfile
```

Then run:


and:


Only use pnpm if the project actually uses pnpm.

### If `yarn.lock` exists

Use yarn instead:

```yml

  run: yarn install --frozen-lockfile
```

Only use yarn if the project actually uses yarn.

## GitHub Actions Workflow

Use this as the base workflow.

Adjust the output file globs after checking the actual project build output.

```yml
name: Build Desktop Release

on:
  push:
    tags:
      - "v*"

permissions:
  contents: write

jobs:
  build-windows:
    name: Build Windows
    runs-on: windows-latest

    steps:
      - name: Checkout repo
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Build Windows desktop app
        run: npm run desktop:dist:win

      - name: Upload Windows release files
        uses: softprops/action-gh-release@v2
        with:
          files: |
            dist/**/*.exe
            dist/**/*.msi
            dist/**/*.zip
            dist/**/*.yml
            dist/**/*.blockmap
            release/**/*.exe
            release/**/*.msi
            release/**/*.zip
            release/**/*.yml
            release/**/*.blockmap
            out/**/*.exe
            out/**/*.msi
            out/**/*.zip
            out/**/*.yml
            out/**/*.blockmap

  build-mac:
    name: Build macOS
    runs-on: macos-latest

    steps:
      - name: Checkout repo
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Build macOS desktop app
        run: npm run desktop:dist:mac

      - name: Upload macOS release files
        uses: softprops/action-gh-release@v2
        with:
          files: |
            dist/**/*.dmg
            dist/**/*.pkg
            dist/**/*.zip
            dist/**/*.yml
            dist/**/*.blockmap
            release/**/*.dmg
            release/**/*.pkg
            release/**/*.zip
            release/**/*.yml
            release/**/*.blockmap
            out/**/*.dmg
            out/**/*.pkg
            out/**/*.zip
            out/**/*.yml
            out/**/*.blockmap
```

## How Releases Work

The user does not upload the compiled app.

The user pushes source code normally.

Then the user creates and pushes a git tag.

Example:

```bash
git add .
git commit -m "add desktop release workflow"
git push
```

Then update the package version using the actual next project version.

Example:

```bash
npm version patch --no-git-tag-version
```

Then commit the version change:

```bash
git add package.json package-lock.json
git commit -m "release v1.0.1"
```

Then create a tag matching the package version:

```bash
git tag v1.0.1
git push
git push origin v1.0.1
```

Replace `v1.0.1` with the real package version.

When the tag is pushed, GitHub Actions runs automatically.

---

## Simple Explanation For The User

Yes, the main thing that triggers the release build is pushing a git tag.

Normal push:

```bash
git push
```

does not make a release.

Tag push:

```bash
git push origin v1.0.1
```

does make a release, because the workflow listens for:

```yml
on:
  push:
    tags:
      - "v*"
```

That means:

```txt
v1.0.0
v1.0.1
v2.0.0
```

will trigger the build.

## Acceptance Criteria

The task is done when:

- `.github/workflows/release.yml` exists
- the workflow triggers on tags starting with `v`
- Windows build uses `npm run desktop:dist:win`
- macOS build uses `npm run desktop:dist:mac`
- existing `desktop:dist:win` and `desktop:dist:mac` scripts are not modified
- dependency installation matches the real lockfile
- output globs match the real build output
- pushing a tag creates a GitHub Release
- release assets include both Windows and macOS builds
- no compiled app files are committed to the repo
